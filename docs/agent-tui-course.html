What is agent-tui?

A terminal that talks to AI coding agents — with safety guardrails built in

Your AI coding assistant,
in the terminal

agent-tui is a Text-based User Interface that lets you chat with an AI agent that can read your files, run shell commands, and help you code — all without leaving your terminal.

Why use a TUI instead of a web interface?

⌨️

No browser needed

Works entirely in your terminal — perfect for server environments or SSH connections

⚡

Keyboard-first

Fast navigation without touching a mouse — exactly how developers like it

🛡️

Built-in safety

Every file operation and shell command requires your approval before running

1
Read your files

Ask it to explain a codebase, debug an error, or review code changes

2
Write and edit files

Have it create new features, refactor code, or fix bugs directly

3

Execute build tools, tests, git commands — all with your approval

AgentProtocol — the contract

The protocol defines what any agent must be able to do. Here's the core of it:

CODE 
class AgentProtocol(Protocol):
  async def stream(message, thread_id):
    yield AgentEvent(...)

  async def approve_tool(tool_id, approved):
    ...

  async def cancel():
    ...
PLAIN ENGLISH

stream() — Takes your message, spits out events like "here's some text" or "I want to run a tool"

approve_tool() — Called when you click Yes/No on an approval widget

cancel() — Stops whatever the agent is currently doing

AgentEvent — what messages look like

Agents communicate with the TUI using typed events. Each event has a type and relevant data:

📝
MESSAGE_CHUNK

A piece of text the agent is streaming to you — appears letter by letter

🔧
TOOL_CALL

The agent wants to run a file operation or shell command — needs your approval

❓
ASK_USER

The agent needs clarification on something before continuing

⚠️
ERROR

Something went wrong — the agent will try to recover

📖
A new vocabulary word

In programming, a StrEnum is just an enum where all values are strings. EventType uses this so event names are human-readable in logs and code.

The AgentAdapter — the translator

The adapter's job is to listen to events from the agent and call the right TUI methods. Here's the core dispatch logic:

CODE 
async def _dispatch(event):
  match event.type:
    case EventType.MESSAGE_CHUNK:
      app.append_assistant_text(event.text)

    case EventType.TOOL_CALL:
      approved = await app.request_tool_approval(...)
      await agent.approve_tool(tool_id, approved)

    case EventType.TOOL_RESULT:
      app.show_tool_result(...)
PLAIN ENGLISH

When a MESSAGE_CHUNK comes in, append text to the chat display

When a TOOL_CALL comes in, ask the user for approval, then tell the agent what they chose

When a TOOL_RESULT comes in, show the result in the UI

💡
Why this separation matters

The adapter doesn't know or care HOW the TUI shows things — it just calls methods. This means you could swap out the entire UI (say, for a web interface) without touching the agent code at all.

The Approval System

How agent-tui keeps dangerous operations under your control

Why approval matters

An AI agent that can read files, write code, and run shell commands is powerful — and potentially dangerous. The approval system puts YOU in control.

👁️

You see what's coming

Before any file is touched or command runs, you see exactly what will happen

✋

You choose

Approve safe operations, deny suspicious ones, or auto-approve with Ctrl+T

🔄

Agent waits

The agent pauses until you decide — it can't proceed without your say-so

What the adapter checks before showing the approval

Before the approval widget even appears, the adapter does safety checks:

1
Is it a file tool?

Check if the path is within allowed directories

2
Is it a shell command?

Check for dangerous patterns like $(command) or $VAR

3
Show approval widget

If checks pass, show the widget and wait for user input

The approval flow in code

CODE 
case EventType.TOOL_CALL:
  # Check for dangerous shell patterns
  if event.tool_name == "execute":
    if contains_dangerous_patterns(command):
      await agent.approve_tool(tool_id, False)
      app.show_error("Command blocked")
      return

  # Request approval from user
  approved = await app.request_tool_approval(...)
  await agent.approve_tool(tool_id, approved)
PLAIN ENGLISH

For execute (shell) commands, check if the command contains dangerous patterns first

If dangerous patterns are found, auto-reject and show an error message

If it's safe, show the approval widget and wait for the user

Forward the user's decision (approved/denied) back to the agent

⚠️
The approval is a speed bump, not a wall

Once you approve a tool, the agent can use it multiple times. Use auto-approve (Ctrl+T) wisely — it's meant for trusted code review workflows where you've already inspected what the agent is doing.

Shell Safety

The dangerous patterns that get blocked before you even see them

Why shell commands are scary

Shell commands can do almost anything — delete files, exfiltrate data, install malware. Even a "safe" command like cat can be weaponized. Here's what agent-tui blocks:

# Command substitution — lets commands hide inside commands $(rm -rf /) `cat /etc/passwd` # Variable expansion — can inject arbitrary commands $HOME ${PATH} $((1+1)) # Process substitution — runs commands in a subshell <(some command) >(some command) # Redirections — can overwrite or append to files >> > < << # Newlines and special characters — can break out of commands \n \r \t

The pattern checker

The regex that detects these dangerous patterns:

CODE 
DANGEROUS_SHELL_PATTERNS = [
    r'\$',      # $ at start of variable
    r'\`',      # backticks
    r'\$(',     # $(command)
    r'<>',     # process substitution
    r'>>',     # append redirect
    r'\s>>\s',  # space + >>
    r'<<<',    # here string
]
PLAIN ENGLISH

Any $ character — could be variable expansion

Backticks — old-style command substitution

$(...) — command substitution syntax

<> — process substitution syntax

>> or any redirect operator

Safe commands that DO work

Not all shell commands are blocked. Simple commands without special characters are fine:

✓

echo hello

Prints text to the terminal

✓

ls -la

Lists files with details

✓

git status

Shows git repository state

✓

pwd

Prints working directory

🛡️
Defense in depth

The dangerous pattern check happens BEFORE the approval widget. This means even if you're not paying attention, basic malicious patterns are blocked automatically. You only see the widget for commands that pass this first check.

Giving the Agent Eyes

Web search and URL fetching — Phase 4

Two new tools enter the arena

Phase 4 adds two LangChain tools to the agent's toolkit:

🔍

web_search

Calls the Tavily API with a query string. Returns a JSON list of results — each with title, URL, and a content snippet.

🌐

fetch_url

Downloads a URL with httpx, converts the HTML to Markdown via markdownify, and truncates to 10,000 chars — so the agent can read web pages as text.

🔑
Lazy credential injection

Both tools read their API keys lazily — TAVILY_API_KEY is written into os.environ only when the agent first runs, not at import time. This means you can configure the TUI without importing the tools.

Why return JSON, not plain text?

Both tools return a JSON string — not a human-readable sentence. This is intentional.

1
Agent sees structured data

The LLM can parse JSON and extract specific fields — title, URL, content — rather than guessing from prose.

2
TUI widget recognises the format

ToolCallMessage._format_web_output() detects JSON with a results key and renders each result with a bold title and a dim URL — much richer than raw text.

3
No double-formatting

The widget is the single source of truth for display logic. The tool just provides the data; the widget decides how to show it. A single responsibility each.

The web search flow

1
Agent emits on_tool_start with name="web_search"

EventTranslator sees a normal tool name (not "task" or "compact_conversation"), so it yields a TOOL_CALL event.

2
Adapter dispatches → approval widget

The TUI shows the tool name and arguments. You decide: approve or deny.

3
Approved → Tavily executes → on_tool_end

Results come back as JSON. EventTranslator yields TOOL_RESULT. The widget formats and displays them.

Agents Talking to Agents

Subagent spawning and planning events — Phase 5

The "task" tool is special

When a DeepAgents agent needs to delegate work, it calls a tool named task. The TUI intercepts this before it ever reaches the approval widget.

🧠
The interception pattern

In EventTranslator._handle_tool_start(), the tool name is checked before the generic TOOL_CALL path. If the name is "task", it yields SUBAGENT_START and returns early. The approval flow never fires.

📋

PLAN_STEP

Agent is decomposing a task. Shows ⚙ Planning N/M with the step text below it in the chat view.

→

SUBAGENT_START

A subagent was spawned. Shows → Subagent: <name> in the chat. Name comes from the description field, truncated to 80 chars.

✓

SUBAGENT_END

Subagent finished. Shows ✓ Subagent done: <name>. Emitted on on_tool_end with name="task", even when there's no output key.

Order matters: check the tool name first

A subtle bug was caught during implementation. Here's why the tool name check must happen before the output-presence guard:

✗
Wrong order (original)

Check if "output" in data → if missing, return early. Then check tool name. Result: if a task tool ends with no output key, SUBAGENT_END is silently dropped.

✓
Right order (fixed)

Extract tool name first. If it's "task", yield SUBAGENT_END and return — regardless of whether output is present. The output-presence guard only applies to regular tools.

💡
The lesson

When you have special cases that bypass generic logic, the special case check must come first. Early-return patterns are powerful — but only when the conditions are evaluated in the right order.

The subagent lifecycle in the TUI

1
Agent emits on_tool_start, name="task"

EventTranslator extracts the description (or task, or "subagent" as fallback), truncates to 80 chars, yields SUBAGENT_START.

2
Adapter dispatches → app.show_subagent_started(name)

Status bar shows "Running subagent: <name>". Chat shows "→ Subagent: <name>" as an inline message.

3
Subagent completes → on_tool_end, name="task"

EventTranslator yields SUBAGENT_END. Adapter calls show_subagent_finished. Status bar resets to "thinking". Chat shows "✓ Subagent done".

Never Lose the Thread

Context compaction, token counting, and memory management — Phase 6

The context window problem

Every LLM has a fixed context window. Long conversations fill it up. DeepAgents handles this automatically by calling a special tool — compact_conversation — to summarise old history. Phase 6 makes this visible in the TUI.

⏳

STATUS_UPDATE

Emitted when compact_conversation starts. Shows "Compacting context…" in the status bar so the user knows to wait.

✓

CONTEXT_SUMMARIZED

Emitted when compaction ends. Carries the remaining token count parsed from the tool's JSON output. Mounts a SummarizationMessage in the chat and updates the status bar.

🔍
Same interception pattern as subagents

Just like the task tool bypasses the approval flow, compact_conversation is intercepted before reaching TOOL_CALL. The check runs before the task check, which runs before the generic path.

Parsing the token count safely

The compaction tool returns JSON, but the token count key isn't standardised. Three possible keys are tried in priority order:

1
tokens_remaining — checked first

The primary key. If present, its value is used — even if it's 0.

2
token_count — fallback

Tried only if tokens_remaining is absent from the dict.

3
remaining_tokens — last resort

If none of the keys are present, token_count defaults to 0. Non-JSON output also yields 0.

⚠️
Key presence, not truthiness

The lookup uses a for key in (...): if key in parsed: break loop — not Python's or chaining. This ensures a value of 0 on the first key doesn't fall through to the second key (since 0 or ... would be truthy-false and continue the chain).

Token percentage in the status bar

Once the context limit is known, the status bar shows both the absolute count and the percentage used:

12K
Without context limit

Shows 12.3K tokens — just the count.

45%
With context limit known

Shows 12.3K (45%) tokens — percentage helps you judge how close to the limit you are before compaction triggers.

📊
Stored in SessionStats

context_limit and context_compaction_count are tracked across turns so you can see the full session history.
